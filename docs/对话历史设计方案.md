# 对话历史设计方案

## 1. 概述

本项目的 AI 对话功能支持**多会话管理**、**消息持久化**、**富内容（图表/建议）附加**。用户与 AI 的每一次对话被组织为一个 **Session（会话）**，会话内包含多条 **Message（消息）**。消息除了文本，还可携带图表配置、数据、推荐问题等结构化数据。

---

## 2. 数据模型（SQLite）

### 2.1 `chat_sessions` — 会话表

| 列 | 类型 | 说明 |
|---|---|---|
| `id` | TEXT (PK) | UUID，由前端或后端生成 |
| `user_id` | INTEGER (FK → users) | 所属用户 |
| `title` | TEXT | 会话标题，默认"新对话"，首条消息后自动截取前 30 字 |
| `mode` | TEXT | 模式：`chart`（图表分析） / `rule`（规则推荐） |
| `created_at` | TEXT | 创建时间，`datetime('now','localtime')` |
| `updated_at` | TEXT | 最后更新时间，每次新增消息自动刷新 |

### 2.2 `chat_messages` — 消息表

| 列 | 类型 | 说明 |
|---|---|---|
| `id` | INTEGER (PK, AUTOINCREMENT) | 自增 ID |
| `session_id` | TEXT (FK → chat_sessions, ON DELETE CASCADE) | 所属会话 |
| `role` | TEXT | `user` 或 `assistant` |
| `content` | TEXT | 消息文本内容 |
| `charts_json` | TEXT (JSON, nullable) | 图表列表 JSON，assistant 消息携带 |
| `suggestions_json` | TEXT (JSON, nullable) | AI 推荐问题列表 JSON |
| `created_at` | TEXT | 创建时间 |

**关键约束：**
- `chat_messages.session_id` 外键级联删除 — 删除会话时自动删除所有消息
- `idx_messages_session` 索引加速按会话查询
- 消息按 `id ASC` 顺序读取，保证时序正确

## 3. 后端架构

### 3.1 数据访问层 — `backend/core/chat_db.py`

提供对 `chat_sessions` 和 `chat_messages` 的全部 CRUD 操作：

```python
# 会话操作
list_sessions(user_id)       # 按 updated_at DESC 列出用户会话列表
get_session(session_id)      # 获取单个会话
create_session(id, user_id, title, mode)   # 创建会话
update_session_title(id, title)            # 更新标题
update_session_mode(id, mode)              # 更新模式
delete_session(id)          # 删除会话（CASCADE 删除消息）

# 消息操作
get_messages(session_id)    # 按 id ASC 获取会话所有消息
add_message(session_id, role, content, charts, suggestions)  # 添加消息
```

`get_messages` 会自动将 `charts_json` / `suggestions_json` 解析为 Python 对象并剥离原始 JSON 字段。`add_message` 会自动更新会话的 `updated_at`。

### 3.2 路由层

#### `POST /api/chat` — `backend/routers/api_chat.py`

核心对话接口，数据流如下：

```
用户输入 → 会话管理 → 加载历史 → 调用 Agent → 保存消息 → 返回回复
```

详细流程：
1. **会话管理**：`session_id` 为空则自动生成 UUID；若传了 `user_id` 则开启 DB 持久化模式
2. **加载历史**：从 `chat_messages` 读取该会话全部消息（截断最近 20 条防 Token 溢出），构造 history 列表传给 Agent
3. **调用 Agent**：`process_chat(message, history)` 经 LangGraph 意图识别→分析→校验→执行
4. **保存消息**：同时写入用户消息和 AI 回复（含 charts / suggestions）
5. **自动标题**：如果是用户的首条消息，截取前 30 字作为会话标题
6. **内存模式**：不传 `user_id` 则不持久化，仅做一次对话

**响应结构 (`ChatResponse`)：**

```json
{
  "reply": "AI 回复文本",
  "charts": [{ "title": "...", "pivot_config": {...}, "data": [...], "chart_type": "bar", "vega_spec": {...} }],
  "suggestions": ["追问1", "追问2"],
  "session_id": "uuid-string",
  "execution_time_ms": 1234
}
```

#### `GET/POST/PUT/DELETE /api/chat/sessions` — `backend/routers/api_sessions.py`

完整的会话生命周期管理：

| 端点 | 功能 |
|---|---|
| `GET /api/chat/sessions?user_id=1` | 获取用户的所有会话 |
| `POST /api/chat/sessions` | 创建新会话（自动生成 UUID） |
| `PUT /api/chat/sessions/{id}` | 重命名会话标题 |
| `DELETE /api/chat/sessions/{id}` | 删除会话及所有消息 |
| `GET /api/chat/sessions/{id}/messages` | 获取会话的消息列表 |

## 4. 前端架构

### 4.1 状态管理 — `frontend/src/stores/useChatStore.ts`

基于 **Pinia Composition API** 实现，采用 **后端持久化 + 用户体系** 模式。

**核心状态：**

```typescript
sessions: ChatSession[]     // 会话列表（仅元数据，消息按需加载）
activeSessionId: string     // 当前活跃会话 ID
loading: boolean            // 发送请求中
user: {id, username, role}  // 当前用户
userList: [...]             // 可用用户列表
```

**核心方法：**

| 方法 | 职责 |
|---|---|
| `init()` | 初始化：加载用户列表 → 自动选择用户 → 拉取会话列表 → 若无会话则创建 |
| `fetchSessions()` | 从后端拉取用户的所有会话（不含消息体） |
| `createSession()` | 后端创建新会话，本地添加到列表头部并激活；复用空会话避免重复创建 |
| `switchSession(id)` | 切换活跃会话，若本地无消息则懒加载 `fetchMessages()` |
| `fetchMessages(sid)` | 从后端拉取指定会话的消息列表 |
| `deleteSession(id)` | 后端删除 + 本地移除 + 自动切换到第一个可用会话 |
| `sendMessage(msg)` | 发消息：乐观推入 user 消息 → POST `/api/chat` → 收到响应后推入 assistant 消息 |
| `clearAllSessions()` | 批量删除所有会话并新建一个 |

**关键设计点：**
- **懒加载消息**：会话列表只返回元数据，具体消息在切换到该会话时才请求
- **乐观更新**：用户消息立即推入本地数组，无需等网络返回
- **自动标题**：前端在发送首条消息时同步更新本地标题，后端也会做同样处理
- **错误处理**：网络失败时推入错误消息，不阻塞 UI

### 4.2 前端类型定义

```typescript
interface ChatMessage {
  role: 'user' | 'assistant'
  content: string
  charts?: Record<string, any>[] | null    // 图表配置+数据
  suggestions?: string[]                    // 推荐追问
}

interface ChatSession {
  id: string
  title: string
  messages: ChatMessage[]                   // 消息列表
  mode: 'chart' | 'rule'
  createdAt: string
  updatedAt: string
}
```

### 4.3 UI 组件

#### `AIDialog.vue` — 全屏 AI 对话面板（主组件）

布局：`侧边栏（全屏模式）` + `消息主区` + `输入区`

- **侧边栏**：显示全部会话列表，支持切换、删除、新建
- **消息区**：气泡式渲染 user/assistant 消息；assistant 消息内可渲染图表卡片（VegaLite）、建议 Chip
- **输入区**：自适应高度 Textarea + 发送按钮
- **弹窗**：图表数据查看、配置/SQL 查看、保存到看板、用户设置
- **快捷键**：Enter 发送，无 Shift 修饰

支持两种交互形态：
1. **全屏模式**：`position: fixed` 铺满视口，显示侧边栏
2. **面板模式**：在 `el-dialog` 内渲染，通过顶栏图标切换历史下拉

#### `ChatBox.vue` — 轻量对话面板（旧版/辅助组件）

简化版对话面板，仅支持纯文本 + 单个 Vega-Lite 图表渲染，无会话管理，适用于嵌入式场景。

## 5. 数据流全景

```
用户输入文本
    │
    ▼
useChatStore.sendMessage()           ← 乐观推入 user 消息
    │
    ▼
POST /api/chat                      ← 携带 session_id + user_id
    │
    ├─ chat_db.get_messages()        ← 加载历史（最近 20 条）
    ├─ process_chat()                 ← LangGraph Agent 处理
    ├─ chat_db.add_message(user)     ← 持久化用户消息
    ├─ chat_db.add_message(assistant)← 持久化 AI 回复（含 charts/suggestions）
    └─ 返回 ChatResponse
    │
    ▼
useChatStore 推入 assistant 消息     ← 更新本地 messages 数组
    │
    ▼
AIDialog.vue 响应式渲染              ← 气泡、图表卡片、建议 Chip
```

## 6. 设计要点总结

| 维度 | 方案 |
|---|---|
| **存储** | SQLite，`chat_sessions` + `chat_messages` 两张表，JSON 列存储结构化数据 |
| **会话管理** | UUID 标识，按用户隔离，`updated_at` DESC 排序 |
| **消息加载** | 按需懒加载，会话列表仅含元数据，切会话时按 `id ASC` 拉取完整消息 |
| **历史截断** | 传给 Agent 的历史截断为最近 20 条，控制 Token 消耗 |
| **自动标题** | 首条用户消息前 30 字作为会话标题 |
| **乐观更新** | 用户消息立即推入前端列表，不依赖网络 RTT |
| **级联删除** | 删除会话 → CASCADE 删除所有消息 |
| **富内容** | charts_json / suggestions_json 作为消息附加属性，解析/序列化在 DB 层自动完成 |
| **双模式** | `chart`（图表生成） / `rule`（规则推荐），通过 session.mode 区分 |
| **无状态模式** | 不传 `user_id` 时使用内存对话，不持久化，适合快速试用 |
| **UI 适配** | 全屏模式侧边栏管理会话，面板模式下拉选择历史，兼顾深度使用与轻量交互 |
