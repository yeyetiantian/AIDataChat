# 企业级数据透视分析系统 - 完整项目提示词（最终版 v3.0）

---

## 1. 项目背景与核心目标

我需要构建一个 Web 版的数据透视分析系统，**功能对标 Tableau/Excel 透视表**，以**图表可视化为主、数据表格为辅**。后端连接一个已有的 **DuckDB 数据库**（`vcloud_duck.db`，7.3 GB），包含 6 张表，总数据量约 **2.95 亿行**。

**核心交互方式**：
1. **拖拽式**：用户将字段拖拽到四个区域（筛选器、轴、图例、值），实时生成图表
2. **AI 对话式**：用户通过自然语言描述分析需求，系统自动生成图表配置

**前端渲染**：自研基于 **Vega-Lite** 的图表渲染组件，配合 `vue-draggable-plus` 实现四象限拖拽交互。

**需要实现功能**：
1、用户自然语言对话生成 透视表配置，调用后端配置生成vega图表渲染配置的接口返回给前端
2、用户拖拽数据字段 生成 透视表配置，后端通过查询配置生成sql语句并基于大宽表查询数据，生成vega图表渲染配置返回给前端
3、用户生成的图表可以保存到看板，在看板可以二次进入配置编辑图表

## 3. 🏗️ 核心架构决策：启动时构建物理大宽表

### 3.2 创建策略：启动时检测并构建

**策略**：系统启动时检测大宽表是否存在，若不存在则自动构建，若存在则直接复用。

- **触发时机**：FastAPI 启动事件（`lifespan` 或 `startup`）
- **执行方式**：后台线程执行，不阻塞 API 服务启动
- **检测逻辑**：查询 `information_schema.tables` 判断表是否存在
- **幂等性**：多次启动仅首次执行构建

### 3.3 大宽表字段范围

**大宽表（WIDE_DETAIL）采用"固定业务字段 + 动态信号列"结构**：

#### 固定数据字段（8 个）
| 列名 | 中文名 | 类型 | 数据来源 |
|------|--------|------|----------|
| `person` | 人员 | VARCHAR | TM_RMU_PS_TASK.CREATE_BY / CREATE_BY_CODE |
| `vehicle_type` | 车型 | VARCHAR | TM_RMU_PS_TASK_VEHICLETYPE.VEHICLETYPE_NAME |
| `vehicle` | 车辆(VIN) | VARCHAR | TL_RMU_PS_TASK_RULE_RESULT.VIN |
| `task` | 任务名称 | VARCHAR | TM_RMU_PS_TASK.TASK_NAME |
| `rule_name` | 规则名称 | VARCHAR | TM_RMU_PS_TASK_RULE.RULE_NAME |
| `rule_type` | 规则类型 | VARCHAR | TM_RMU_PS_TASK_RULE.RULE_TYPE→映射('统计'/'报警'/'事件') |
| `alarm_time` | 报警时间 | TIMESTAMP | TL_RMU_PS_TASK_RULE_RESULT.TRIGGER_TIME |
| `duration_sec` | 持续时间(秒) | DOUBLE | EXIT_TIME - TRIGGER_TIME |

#### 动态信号列（TOP-N 平铺）
按信号表 `TL_RMU_PS_TASK_RULE_RESULT_SIGNAL` 中 SIGNAL_NAME 出现频率最高的 TOP-N（默认 200 个）平铺为列，列名即信号名（如 `UDat_13`、`PrplsnSysAtv`、`IBSBatSOC` 等），值为 SIGNAL_VALUE。

#### 构建逻辑
```sql
-- 固定 8 字段由 5 表 LEFT JOIN 得到（不经过信号表）
-- 信号列通过 PIVOT（CASE WHEN 聚合 或 DuckDB PIVOT）将 SIGNAL_NAME→列名、VALUE→列值 展开
-- 大宽表构建时机：FastAPI 启动时后台线程执行（同原策略）
```

### 3.4 索引策略

为加速查询，在以下高频过滤字段上创建索引：
- `TRIGGER_TIME`（时间范围过滤）
- `TASK_ID`（任务筛选）
- `VIN`（车辆筛选）
- `RULE_NAME`（规则筛选）
- `VEHICLETYPE_NAME`（车型筛选）

### 3.5 构建时机

- **主要时机**：FastAPI 启动时后台线程执行
- **手动刷新**：提供管理接口 `/api/admin/refresh_wide_detail`，允许手动触发重建
- **无查询时兜底**：所有查询均假设 WIDE_DETAIL 表已存在或通过 CTE 实时构建

## 4. 前端方案：Vega-Lite + 自研拖拽组件

### 4.1 整体架构

基于 **Vega-Lite** 自研实现，包含：

- **拖拽区**：使用 `vue-draggable-plus` 实现字段拖拽，筛选器、轴、图例、值
- **Vega-Lite 图表渲染**：使用 `vega-embed` 或 `vue-vega` 将后端返回的数据 + 配置渲染为图表
- **数据表格**：作为辅助功能，仅在用户点击"查看数据"时以弹窗/抽屉形式展示
- 配置区域需要额外配置duckdb支持的额外配置语法，比如having、sort、limit等等

### 4.2 数据流

```
用户拖拽字段到四象限 → Pinia Store 更新配置 
    → 调用 FastAPI /api/pivot（传入 JSON 配置）
    → 后端返回 { data: 聚合结果, config: 回显配置, vega_spec: 图表规格 }
    → 前端提取 data 和 vega_spec，调用 vega-embed 渲染图表
    → 用户点击图表数据点 → 弹出明细数据表格
```

### 4.3 优势

- 完全自研，UI 完全可控
- 无第三方图表组件依赖，包体积更小
- Vega-Lite 规范灵活，可自由定制图表类型和样式
- 与后端 JSON 配置天然对齐，无需额外转换层

## 5. 核心数据模型（严格四属性 JSON）

```json
{
  // 1. 筛选器（Filters）→ SQL WHERE
  "filters": [
    { "field": "TRIGGER_TIME", "op": "between", "value": ["2026-07-01", "2026-07-04"] }
  ],
  // 支持操作符：=, !=, >, >=, <, <=, between, in, like, contains, starts_with, ends_with, is_null, is_not_null

  // 2. 轴（Axes）→ SQL GROUP BY
  "axes": [
    { "field": "TRIGGER_TIME", "alias": "日期", "group": "month", "sort": "asc" }
  ],

  // 3. 图例（Legend）→ SQL PIVOT ON
  "legend": [
    { "field": "VEHICLETYPE_NAME", "alias": "车型" }
  ],

  // 4. 值（Values）→ SQL 聚合
  "values": [
    {
      "id": "val_1",
      "field": "TASK_RULE_RESULT_ID",
      "aggregation": "count",
      "alias": "触发次数",
      "expr": null,  // 可选，与 field+aggregation 互斥，支持直接写 DuckDB 表达式
      "show_as": { "type": "column_percentage" }
    }
  ],
  // 扩展属性
  "having": [],
  "order_by": [
    { "field": "触发次数", "direction": "desc" }
  ],
  "pagination": { "page": 1, "pageSize": 100 },
  "grand_total": true,
  "subtotals": true,
  "limit": 10000
}
```

## 6. 后端 SQL 生成策略（基于明细宽表）

### 6.1 核心原则

- **数据来源**：明细宽表 `WIDE_DETAIL`（物理物化表），或 Pivot 场景中以 CTE 实时构建（8 固定字段 + 动态信号列）
- **固定字段**：`person`、`vehicle_type`、`vehicle`、`task`、`rule_name`、`rule_type`、`alarm_time`、`duration_sec`，可直接引用
- **动态信号列**：按 SIGNAL_NAME 出现频率 TOP-N 展开为列，列名即信号名（如 `UDat_13`、`PrplsnSysAtv`），信号值为 VARCHAR 类型，数值聚合时需 `TRY_CAST` 转换

### 6.2 SQL 生成流水线

```
┌─────────────────────────────────────────────────────────────────────┐
│ 步骤 1：构建数据源（Base CTE）                                     │
│   → WITH _wide_base: 8 固定字段 JOIN（5 表 LEFT JOIN）             │
│   → WITH _wide_sig:   关联信号表，筛选目标信号名                    │
│   → WITH pivoted:    PIVOT 展开信号（SIGNAL_NAME→列名）            │
│   → SELECT 固定列 + 信号列 FROM _wide_base LEFT JOIN pivoted       │
│   → WHERE（来自 filters）                                           │
│   → 输出：base_data CTE                                           │
├─────────────────────────────────────────────────────────────────────┤
│ 步骤 2：分组聚合（含 PIVOT 判断）                                  │
│   ├─ 若有 legend：                                                 │
│   │    → PIVOT base_data                                          │
│   │    → ON legend_field                                          │
│   │    → USING aggregations / expr                                │
│   │    → GROUP BY axes                                            │
│   │    → 若 row_filters 筛选维度字段 → HAVING                    │
│   │    → 输出：pivot_data CTE                                     │
│   └─ 若无 legend：                                                 │
│        → SELECT axes + aggregations / expr                        │
│        → GROUP BY axes                                            │
│        → 若 row_filters 筛选维度字段 → HAVING                    │
│        → 输出：grouped_data CTE                                   │
├─────────────────────────────────────────────────────────────────────┤
│ 步骤 3-9：动态列名 → Show As → HAVING → 总计小计 → TOP N → 排序 → 分页│
└─────────────────────────────────────────────────────────────────────┘
```

### 6.3 值显示方式 SQL 模板

| show_as.type           | SQL 实现                                       |
| :--------------------- | :--------------------------------------------- |
| `column_percentage`    | `col / SUM(col) OVER ()`                       |
| `row_percentage`       | `col / SUM(col) OVER (PARTITION BY row_field)` |
| `total_percentage`     | `col / SUM(col) OVER ()`                       |
| `difference`           | `col - (基准列值)`                             |
| `running_total`        | `SUM(col) OVER (ORDER BY running_field)`       |
| `rank_asc`/`rank_desc` | `RANK() OVER (ORDER BY col ASC/DESC)`          |

### 6.4 expr 表达式支持

- `values[].expr` 与 `field` + `aggregation` 互斥
- 若有 `expr`，直接将其作为 DuckDB 表达式嵌入 `USING` 子句
- 支持复杂运算：`sum(TASK_RULE_RESULT_ID) / count(distinct VEHICLE_SID)`
